## 二、技术架构梳理

### 2.1 当前架构（分层）

```
┌─────────────────────────────────────────────────────────┐
│  表现层  main.py → Game.draw()、village/combat UI       │
├─────────────────────────────────────────────────────────┤
│  场景层  village | combat（game.py 内嵌，未用 Scene 基类）│
├─────────────────────────────────────────────────────────┤
│  节点层  nodes/combat、shop（部分接入，game.py 主控）     │
├─────────────────────────────────────────────────────────┤
│  系统层  systems/route、COMBAT_REWARD_POOL               │
├─────────────────────────────────────────────────────────┤
│  核心层  core/EventBus、GameState、SceneManager          │
├─────────────────────────────────────────────────────────┤
│  数据层  levels.py、config、setting、unlock、meta、data/  │
└─────────────────────────────────────────────────────────┘
```

### 2.2 核心模块职责

| 模块 | 职责 | 状态 |
|------|------|------|
| `game.py` | 主循环、村子/战斗切换、关卡加载、路线、商店、结算 | 核心，约 1800 行，偏重 |
| `enemy.py` | 6 类型 + hitrun、投射物、AOE、create_enemy | ✅ 完整 |
| `player.py` | 移动、攻击、灵根/法宝、冲刺、受击 | ✅ 完整 |
| `levels.py` | 关卡配置、大关过滤、路线树 | ✅ 完整 |
| `village.py` | 村子 UI、伙伴房间、灵根/法宝选择 | ✅ 完整 |
| `core/` | EventBus、GameState、SceneManager | ✅ 框架在 |
| `nodes/` | combat、shop 注册，NodeHandler 接口 | ⚠️ 部分接入 |
| `systems/` | RouteSystem | ✅ |

### 2.3 与成熟项目的差距

| 维度 | 当前 | 成熟项目目标 |
|------|------|--------------|
| **代码组织** | game.py 过重，逻辑集中 | 按场景/节点拆分，职责清晰 |
| **配置化** | levels.json、enemies.json 已用 | 更多表驱动，少硬编码 |
| **可测试性** | 无单元测试 | 核心逻辑可单测 |
| **构建/发布** | 手跑 python main.py | 打包脚本、版本号、发布流程 |
| **文档** | 设计文档多，缺架构图 | 架构图、API 说明、贡献指南 |
| **美术管线** | 手动放图、预览工具 | 规范命名、批量导入、图集 |
